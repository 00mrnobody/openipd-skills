## 1. 执行摘要 (Executive Summary)

### 1.1 技术可行性结论
| 评估维度 | 结论 | 置信度 | 备注 |
|---------|------|--------|------|
| 整体技术可行性 | **可行，但有风险** | 75% | 移动底盘和云端依赖是关键风险点 |
| 方案成熟度 | 中等 | 70% | 大部分技术已成熟，部分需定制开发 |
| 技术竞争力 | 中等偏上 | 65% | 差异化在移动能力，但存在实现挑战 |
| 成本可控性 | **高风险** | 60% | 移动底盘BOM占比过高 |

### 1.2 核心发现
- ✅ **云端语音方案**: WebRTC+RTC技术成熟，可行性高
- ⚠️ **移动底盘方案**: 技术可行但成本压力大，安全性需验证
- ✅ **轻量视觉AI**: RK3566/68可支撑边缘推理
- ⚠️ **生态协议**: Matter协议复杂度高，建议分阶段实施
- ❌ **网络降级策略**: 当前需求缺失，必须补充

### 1.3 对CDCP的建议
| 建议项 | 优先级 | 说明 |

### 3.2 桌面移动底盘技术

#### 3.2.1 MKTPDT需求挑战 ⚠️ **重要**

> **SE质疑**: MKTPDT定义的"桌面自主巡航"需求需重新审视

| 挑战点 | 分析 | 风险等级 |
|--------|------|---------|
| **安全性质疑** | 桌面跌落100%可靠不现实，需定义"可接受失败率" | 高 |
| **实用性质疑** | 桌面空间有限，巡航价值vs风险不成比例 | 高 |
| **成本质疑** | 移动底盘BOM占比过高，影响定价竞争力 | 高 |
| **维护性质疑** | 机械结构增加故障点，影响产品可靠性 | 中 |

**建议需求调整**:
```diff
- 原需求: "桌面范围自主巡航" (高优先级)
+ 建议: "定点位置调整" (中优先级)
+ 建议: "跌落检测保护" (P0安全需求)
- 原需求: "SLAM-lite" (高优先级)
+ 建议: "简单定位+防跌落" (中优先级)
```

#### 3.2.2 技术方案对比

| 方案 | 原理 | 成本 | 复杂度 | 适用场景 | 推荐度 |
|------|------|------|--------|---------|--------|
| **双轮差速+陀螺仪** | 传统底盘方案 | ¥80-120 | 中 | 平坦桌面 | ⭐⭐⭐⭐ |
| **麦克纳姆轮全向** | 全向移动 | ¥150-200 | 高 | 复杂桌面 | ⭐⭐⭐ |
| **履带式** | 履带驱动 | ¥100-150 | 中 | 非平坦表面 | ⭐⭐ |
| **单球轮平衡** | 球轮驱动 | ¥200+ | 极高 | 高端产品 | ⭐ |

```python
# 伪代码: 多传感器融合防跌落
class AntiFallSystem:
    def check_fall_risk(self):
        # 红外传感器数据 (4方向)
        ir_data = self.ir_sensors.read()  # [front, back, left, right]
        
        # ToF测距 (垂直向下)
        tof_distance = self.tof_sensor.read()
        
        # IMU姿态
        pitch, roll = self.imu.get_attitude()
        
        # 融合决策
        if tof_distance > FALL_THRESHOLD or any(ir_data < IR_THRESHOLD):
            if abs(pitch) > TILT_THRESHOLD or abs(roll) > TILT_THRESHOLD:
                return CRITICAL  # 立即停车
            return WARNING  # 减速
        return SAFE
```

### 3.3 轻量视觉AI (边缘推理)

#### 3.3.1 RK3566/68 AI能力评估

| 指标 | RK3566 | RK3568 | 说明 |
|------|--------|--------|------|
| CPU | 4×A55@1.8GHz | 4×A55@2.0GHz | NEON SIMD支持 |
| NPU | ❌ 无 | ❌ 无 | 必须使用CPU推理 |
| GPU | Mali-G52 | Mali-G52 | OpenCL支持 |
| 内存带宽 | 支持LPDDR4 | 支持LPDDR4 | 推理瓶颈 |

#### 3.3.2 模型选型与性能预估

| 功能 | 模型选择 | 输入尺寸 | 参数量 | 预估延迟 | 准确率 |
|------|---------|---------|--------|---------|--------|
| 人脸检测 | MobileNet-SSD | 300×300 | 2.3M | 80ms | 92% |
| 人脸识别 | FaceNet-Lite | 160×160 | 1.2M | 120ms | 88% |
| 手势识别 | Custom-CNN | 96×96 | 0.5M | 50ms | 85% |
| 情绪识别 | MobileNet-V2 | 224×224 | 3.5M | 150ms | 82% |

#### 3.3.3 优化策略
```
推理优化方案:
├── 框架: TensorFlow Lite + XNNPACK delegate
├── 量化: INT8量化，模型大小减少75%
├── 多线程: 4线程并行，A55全核心利用
├── 预处理: OpenCV NEON优化
├── 批处理: 单帧处理，无batch
└── 缓存: 人脸特征库本地缓存
```

#### 3.3.4 技术可行性结论
| 评估项 | 结论 | 说明 |
|--------|------|------|
| 人脸检测 | ✅ 可行 | 延迟<100ms，满足实时性 |
| 人脸识别 | ✅ 可行 | 需限制人脸库规模<100人 |
| 手势识别 | ⚠️ 可行 | 准确率目标需下调至85% |

### 7.2 关键指标定义

```yaml
Metric-M01-VoiceLatency:
  name: 端到端语音延迟
  definition: 用户说话结束到听到设备回应的时间间隔
  target: <500ms (P95)
  measurement: 1000次对话采样
  test_method: 自动化测试脚本 + 人工抽检
  owner: RDPDT

  name: 人脸检测准确率

  measurement: COCO-face测试集 + 自建数据集
  test_method: 离线模型评估
  owner: 软件SE

Metric-M03-AntiFall:
  name: 防跌落可靠性
  definition: 正确触发保护的次数 / 接近边缘总次数
  target: >95% (注意: 非100%)
  measurement: 1000次边界测试
  test_method: 自动化测试台
  owner: 硬件SE
```

> - 建议调整为">95%检测率+跌落保护设计"(如边缘缓冲)
